## Как получить логи от заказчика:

### Способ 1: Через Xcode (если есть доступ к Mac)
1. Подключите iPhone заказчика к Mac
2. Откройте Xcode → Window → Devices and Simulators
3. Выберите устройство заказчика
4. Нажмите "Open Console" для просмотра логов в реальном времени
5. Попросите заказчика воспроизвести проблему (попытаться войти)
6. Скопируйте все логи, особенно строки с эмодзи (🔐, 📡, ❌, 💥)

### Способ 2: Через TestFlight Crash Reports (Рекомендуется)

#### Шаг 1: Включить Share Analytics на устройстве клиента

**Инструкция для клиента:**
1. Откройте приложение **TestFlight** на iPhone/iPad
2. Перейдите на вкладку **"Me"** (Я) внизу экрана
3. Найдите раздел **"Share Analytics"** (Поделиться аналитикой)
4. Включите переключатель **"Share Analytics"** (он должен стать зеленым)
5. Подтвердите согласие, если появится диалог

**Важно:** 
- Эта настройка должна быть включена ДО того, как возникнет проблема
- Если проблема уже произошла, включите настройку и попросите клиента воспроизвести проблему снова

#### Шаг 2: Просмотр Crash Reports в App Store Connect

**Для разработчика:**
1. Войдите в [App Store Connect](https://appstoreconnect.apple.com)
2. Выберите **"Мои приложения"** (My Apps)
3. Выберите ваше приложение **"Messenger App"**
4. Перейдите на вкладку **"TestFlight"** в верхнем меню
5. В левом боковом меню найдите раздел **"Crash Reports"** (Отчеты о сбоях)
6. Вы увидите список всех крашей, отсортированных по дате

**Фильтрация отчетов:**
- **По версии:** Выберите версию сборки (например, 1.0.0 (7))
- **По дате:** Используйте календарь для выбора периода
- **По устройству:** Фильтр по модели iPhone (например, iPhone 16 Pro Max)
- **По версии iOS:** Фильтр по версии операционной системы

**Просмотр деталей краша:**
1. Кликните на нужный отчет о краше
2. Вы увидите:
   - **Дату и время** краша
   - **Модель устройства** и версию iOS
   - **Стек трейс** (stack trace) - последовательность вызовов функций, которая привела к крашу
   - **Символизированный стек трейс** (если загружены dSYM файлы)
   - **Процент пользователей**, столкнувшихся с этой проблемой

**Ограничения Crash Reports:**
- ⚠️ Показывают только **краши** (когда приложение полностью закрылось)
- ⚠️ **Не показывают** обычные логи (print/debug сообщения)
- ⚠️ **Не показывают** ошибки, которые не приводят к крашу (например, ошибки регистрации)

**Для получения полных логов** используйте Способ 1 (Xcode) или Способ 4 (встроенное логирование)

### Способ 3: Через TestFlight Feedback
1. Попросите заказчика открыть приложение TestFlight
2. Найти ваше приложение
3. Нажать на "..." (три точки) справа от приложения
4. Выбрать **"Send Feedback"** или **"Report a Problem"**
5. Попросите описать проблему и приложить скриншоты с ошибками
6. Отзывы будут доступны в App Store Connect → TestFlight → Feedback

### Способ 4: Через настройки iPhone (Аналитика)

**Инструкция для клиента:**
1. Откройте **Настройки** (Settings) на iPhone
2. Перейдите в **Конфиденциальность** (Privacy)
3. Выберите **Аналитика и улучшения** (Analytics & Improvements)
4. Нажмите **Данные аналитики** (Analytics Data)
5. Найдите записи с названием вашего приложения (например, "messenger_app" или "Messenger App")
6. Откройте нужную запись (они отсортированы по дате)
7. Скопируйте содержимое (длинное нажатие → Выбрать все → Копировать)
8. Отправьте скопированный текст разработчику

**Что содержит аналитика:**
- Информация о крашах
- Производительность приложения
- Использование памяти
- Ошибки системы

**Ограничения:**
- Данные появляются не сразу (может занять несколько часов)
- Не все логи приложения попадают в аналитику
- Требует ручного копирования от клиента

### Способ 5: Через Console.app на Mac (если есть доступ к Mac клиента)

**Требования:**
- Mac с macOS
- iPhone клиента подключен к Mac через USB
- Доверие между устройствами установлено

**Инструкция:**
1. На Mac откройте приложение **Console** (Консоль)
2. В левой панели выберите подключенный iPhone
3. В поиске введите название приложения или фильтр по процессу
4. Попросите клиента воспроизвести проблему
5. Логи будут отображаться в реальном времени
6. Скопируйте нужные логи (особенно строки с эмодзи: 🔐, 📝, 📡, ❌, 💥)

**Преимущества:**
- ✅ Видны все логи в реальном времени
- ✅ Можно фильтровать по процессу, уровню, времени
- ✅ Самый полный способ получения логов

## Что проверить у заказчика:

### 1. Версия iOS
- Минимальная версия: iOS 14.0
- Рекомендуемая: iOS 15.0+

### 2. Интернет-соединение
- Проверить, работает ли интернет (открыть Safari)
- Попробовать Wi-Fi и мобильный интернет
- Проверить, не блокирует ли корпоративный/публичный Wi-Fi HTTP трафик

### 3. Настройки безопасности
- Настройки → Конфиденциальность → Безопасность
- Проверить, не включены ли какие-то ограничения

### 4. Дата и время
- Настройки → Основные → Дата и время
- Должно быть "Автоматически" включено

### 5. VPN и прокси
- Отключить VPN, если включен
- Проверить настройки прокси

## Типичные ошибки и решения:

### "Сервер не отвечает"
- **Причина**: Проблемы с сетью или сервер недоступен
- **Решение**: Проверить интернет, попробовать другую сеть

### "Таймаут подключения"
- **Причина**: Медленное соединение или сервер перегружен
- **Решение**: Подождать и попробовать снова, проверить скорость интернета

### "Ошибка сети: Failed host lookup"
- **Причина**: Нет подключения к интернету или DNS проблемы
- **Решение**: Проверить интернет, перезагрузить роутер

### "Ошибка обработки ответа сервера"
- **Причина**: Сервер вернул некорректный формат данных
- **Решение**: Проверить логи сервера, связаться с разработчиком бэкенда

## Что делать дальше:

1. **Собрать логи** от заказчика (см. способы выше)
2. **Проверить логи** на наличие ошибок с эмодзи:
   - 🔐 - попытки авторизации/входа
   - 📝 - попытки регистрации
   - 📱 - отправка/проверка OTP кода
   - 📡 - ответы сервера
   - ❌ - ошибки
   - 💥 - исключения и сетевые ошибки
   - ⏱️ - таймауты запросов
3. **Проверить URL** в логах - должен быть `http://83.166.246.225:3000`
4. **Проверить статус код** ответа сервера (200 = успех, 400/500 = ошибка)
5. **Проверить тело ответа** - там может быть детальное сообщение об ошибке от сервера
6. **Связаться с разработчиком бэкенда**, если проблема на стороне сервера

## Детальная информация о TestFlight Crash Reports:

### Когда использовать Crash Reports:
- ✅ Приложение полностью закрылось (краш)
- ✅ Приложение зависло и было закрыто системой
- ✅ Ошибки памяти (out of memory)
- ✅ Необработанные исключения

### Когда Crash Reports НЕ помогут:
- ❌ Ошибки регистрации/авторизации (если приложение не крашится)
- ❌ Проблемы с сетью (если приложение продолжает работать)
- ❌ Ошибки валидации данных
- ❌ Проблемы с UI/UX

### Для таких случаев используйте:
- **Способ 1** (Xcode) - для полных логов
- **Способ 5** (Console.app) - для логов в реальном времени
- **Встроенное логирование** - если реализовано в приложении

### Как интерпретировать Crash Reports:

**Пример стек трейса:**
```
Thread 0 Crashed:
0   libsystem_kernel.dylib        0x0000000181234567 __pthread_kill + 8
1   libsystem_pthread.dylib      0x0000000181345678 pthread_kill + 272
2   libsystem_c.dylib            0x0000000181123456 abort + 104
3   Flutter                       0x0000000101234567 [ваш код]
```

**Что искать:**
- Строки с названием вашего приложения или Flutter
- Упоминания `api_service.dart`, `auth_service.dart`, `auth_provider.dart`
- Ошибки сети (`ClientException`, `SocketException`)
- Ошибки парсинга JSON (`FormatException`)

### Настройка символов (dSYM) для лучшей диагностики:

Чтобы видеть читаемые имена функций вместо адресов памяти:

1. В Xcode: Product → Archive
2. После архивации: Window → Organizer
3. Выберите архив → Distribute App
4. При загрузке в App Store Connect убедитесь, что **"Upload your app's symbols"** включено
5. dSYM файлы будут автоматически загружены и применены к Crash Reports
